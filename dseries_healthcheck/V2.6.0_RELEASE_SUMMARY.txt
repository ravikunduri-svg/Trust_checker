================================================================================

OVERVIEW
================================================================================

Version 2.6.0 introduces database-agnostic SQL query support, expanded health
check coverage, and improved user experience through cleaner error messages
and fixed script compatibility.

================================================================================
KEY ENHANCEMENTS
================================================================================

1. DATABASE-AGNOSTIC SQL QUERIES
   -------------------------------
   - Automatic SQL syntax conversion based on database type
   - Supports PostgreSQL, Oracle, DB2, and SQL Server
   - Handles differences in:
     * String concatenation (|| vs +)
     * Date/time functions (CURRENT_TIMESTAMP vs SYSDATE vs GETDATE())
     * Interval syntax
   
   Technical Implementation:
   - New method: convertQueryForDatabase(query, dbType)
   - Automatically detects database type from JDBC URL
   - Converts queries at runtime before execution
   - No manual query modification needed by users

2. EXPANDED HEALTH CHECK QUERIES
   ------------------------------
   Added comprehensive checks across multiple categories:
   
   AGENT HEALTH (3 new checks):
   - AGENT-001: Total Agent Count
   - AGENT-002: Offline Agents
   - AGENT-003: Agents with Failed Actions
   
   EVENT AND TRIGGER (2 new checks):
   - EVENT-001: Pending Events
   - EVENT-002: Failed Events Last 24 Hours
   
   HIGH AVAILABILITY (2 enhanced checks):
   - HA-001: HA Configuration Status (from ESP_CONFIG_GROUP)
   - HA-002: HA Stage2 Nodes (from ESP_STAGE2HAC)
   
   PERFORMANCE (2 additional checks):
   - PERF-004: Jobs in EXEC Status
   - PERF-005: Jobs Waiting to Run
   
   CALENDAR AND SCHEDULE (2 new checks):
   - CAL-001: Active Calendars
   - CAL-002: Expired Calendars
   
   RESOURCE MANAGER (2 new checks):
   - RM-001: Total Resources
   - RM-002: Resource Conflicts
   
   SECURITY (2 new checks):
   - SEC-001: Total Users
   - SEC-002: Active Sessions

3. IMPROVED ERROR MESSAGES
   -----------------------
   - Removed internal class names (e.g., "Scrambler") from error messages
   - Changed to user-friendly terms:
     * "dSeries encryption" instead of "Scrambler"
     * "dSeries encryption library" instead of "Scrambler class"
     * "Password decryption method" instead of "Scrambler.recover method"
   - Maintains technical accuracy while improving clarity

4. SHELL SCRIPT FIX
   ----------------
   - Removed control-M (carriage return) characters from .sh file
   - Ensures proper execution on Unix/Linux/AIX systems
   - Prevents "bad interpreter" errors

================================================================================
TECHNICAL DETAILS
================================================================================

Database Syntax Conversion Logic:
---------------------------------

SQL Server:
  - String concatenation: || → +
  - CURRENT_TIMESTAMP → GETDATE()
  - INTERVAL 'N hours' → DATEADD(HOUR, -N, GETDATE())

Oracle:
  - CURRENT_TIMESTAMP → SYSDATE
  - INTERVAL 'N hour' → INTERVAL 'N' HOUR

DB2:
  - CURRENT_TIMESTAMP → CURRENT TIMESTAMP
  - INTERVAL 'N hour' → N HOUR

PostgreSQL:
  - No conversion needed (default syntax)

Query Conversion Example:
------------------------
Original (PostgreSQL):
  SELECT COUNT(*) FROM ESP_GENERIC_JOB 
  WHERE START_DATE_TIME < CURRENT_TIMESTAMP - INTERVAL '4 hours'

SQL Server:
  SELECT COUNT(*) FROM ESP_GENERIC_JOB 
  WHERE START_DATE_TIME < DATEADD(HOUR, -4, GETDATE())

Oracle:
  SELECT COUNT(*) FROM ESP_GENERIC_JOB 
  WHERE START_DATE_TIME < SYSDATE - INTERVAL '4' HOUR

================================================================================
KNOWN LIMITATIONS
================================================================================

1. Query conversion is best-effort:
   - Handles common syntax differences
   - Complex database-specific functions may need manual adjustment
   - Test custom queries in target database

2. Some advanced SQL features not converted:
   - Database-specific window functions
   - Proprietary extensions
   - Complex date arithmetic beyond INTERVAL

3. Conversion assumes standard SQL:
   - Non-standard syntax may not convert correctly
   - Recommend testing in development environment first
